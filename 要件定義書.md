# 要件定義書

## 概要

PayPayが出力する利用明細（CSV）を取り込み、Money Forward ME（個人向け）へ自動で記録（登録）するためのWindows向けローカルツール（単一バイナリ）。ユーザー（Shinoryo）自身のPC上で動作し、認証情報はクラウドに保存しない方式を採用する。ドライラン（シミュレーション）モードとフル自動（MFへのアップロード・登録を自動化）を提供する。

## 背景・目的

- **背景**：PayPayの利用明細を目視で確認し、Money Forwardに手動入力しているため工数が大きい。
- **目的**：PayPay CSVを選択するだけでMoney Forwardへの登録を自動化し、入力ミスと作業時間を削減する。
- **成功基準（KPI）**
  - 手動入力工数を70%以上削減。
  - CSV取引の変換精度99%（サンプル100件中99件期待どおり）。
  - 重複登録を発生させない（重複はスキップまたは明示通知）。
  - 「PPCD_A_」で始まる取引（クレジットカード取引）は登録対象外。

## 対象範囲

- **対象データ**：PayPayが出力するCSV（提供されたカラム構成を想定）
- **対象機能**：CSVの読み込み・パース・フィルタ（除外ルール）・カテゴリマッピング・MF用登録の自動実行・ログ／エラー出力
- **実行環境**：Windows 11 上の単一バイナリ（Python実装：Playwright、PyInstallerで配布手順提供）
- **対象ユーザー**：Shinoryo（単独運用）
- **除外**：Money Forward の内部会計ロジックの自動最適化、マルチユーザー運用（初期段階）

## 利用者・関係者

- **主利用者**：Shinoryo（依頼者）
  - 役割：CSV選択 → 実行 → 結果確認・保守
  - 運用制約：他者データにアクセスさせないこと、ユーザー自身が保守すること
- **外部関係者**：なし（個人運用のため）

### 各機能の詳細

#### F01 CSV取り込み

- Shift_JIS / UTF-8 の文字コードに対応する。
- parser.encoding_priority に従い、順番に文字コードを試みる。
- カンマ区切り・ダブルクォートを正しく処理する。

#### F02 CSVパーシング

- 抽出カラム：取引日、出金金額、入金金額、取引先、取引方法、支払い区分、利用者、取引番号。
- 金額文字列（例: "1,280"）を数値（整数）に変換する。
- 複合支払（例: 「PayPayポイント (73円), クレジット VISA 4575 (257円)」）は金額を合算して1件として扱う。

#### F03 除外ルール

- config.yml の exclude_prefixes（初期値: ["PPCD_A_"]）に合致する取引番号の行は処理対象外とする。
- 除外された行はログに記録する（件数および取引番号を出力）。

#### F04 重複検知

- 取引番号が存在する場合はそれを最優先キーとして重複判定を行う。
- 取引番号が欠損している場合は「日時＋金額＋取引先」の組み合わせで重複判定する。
- 重複と判定した行はスキップし、ログにその旨を記録する。

#### F05 MFへの自動登録

- Playwright を使い、Chrome（ユーザー指定のプロファイル）を自動制御する。
- 既存のChromeプロファイルのセッション（Cookie等）を利用して処理し、認証情報をツールに保存しない。
- 自動化操作：MFへの画面遷移・ファイルアップロード・ボタン押下等。

#### F06 マッピング設定

- config.yml の mapping_rules にキーワードとカテゴリのペアを設定する。
- match_mode（"contains" / "starts_with" / "regex"）と priority（優先度）を設定可能。
- 取引先にキーワードが合致した場合、対応するカテゴリを付与する。
- どのルールにも合致しない場合は未分類とする��

#### F07 実行前チェック

- ツール起動時、Chrome プロセスが稼働していないことを確認する。
- Chrome が稼働中の場合は処理を中断し、ユーザーに Chrome を終了するよう促すメッセージを表示する。

#### F08 ドライランモード

- dry_run: true の場合、MFへの書き込みは行わない。
- ブラウザ起動・遷移・操作のシミュレーションは実行し、処理結果をログに記録する。
- 実行開始時に「DRY RUN: no writes performed」とログに出力する。

#### F09 ログ出力

- 実行ログ・エラーCSV・スクリーンショットを log_settings.logs_dir に保存する。
- 実行ログには以下の情報を含む：実行日時・処理件数・除外件数・重複スキップ件数・失敗件数。
- 失敗した行はエラーCSVとして別ファイルに出力する。
- advanced.screenshot_on_error が true の場合、エラー発生時にスクリーンショットを保存する。

#### F10 設定ファイル

- ツールフォルダ直下に config.yml を配置する。
- 起動時に config.yml の存在チェックおよびバリデーションを行う。
- 必須項目（chrome_user_data_dir / chrome_profile / dry_run）が欠落している場合は起動を中止し、詳細エラーをユーザーに通知する。

#### F11 エラーハンドリング

- Playwright の操作失敗時（セレクタ未検出・タイムアウト等）は処理を中断する。
- エラー発生時にスクリーンショットと詳細ログを logs_dir に保存する。
- ユーザーに中断原因と再実行の手順をわかりやすく表示する。

## 非機能要件

| 区分 | 要件 | 優先度 |
| ------ | ------ | -------- |
| セキュリティ | 認証情報をクラウドに保存しない。Cookie/sessionはローカルの Chrome プロファイルを利用 | 必須 |
| セキュリティ | ローカル環境内で動作し、外部へデータ送信を行わない | 必須 |
| セキュリティ | ログに個人情報が含まれるため、ローカル保存を前提とする（暗号化はオプション） | 高 |
| 運用性 | 依頼者単独で保守可能な設計。設定は config.yml の編集で変更できること | 必須 |
| 運用性 | 障害時はログとスクショで原因を特定できること | 必須 |
| 可用性 | オンデマンド実行を想定。常時稼働不要 | 低 |
| コスト | ローカル実行のためクラウド費用は不要 | 必須 |
| 拡張性 | 将来的に別サービスのCSV対応・GUI設定・マッピング学習等を追加可能な設計 | 中 |

## 制約条件

| 制約 | 内容 | 優先度 |
| ------ | ------ | -------- |
| 予算 | 極力コストをかけない（個人運用） | 高 |
| 人的リソース | 運用・保守は依頼者一人 | 必須 |
| セキュリティ | 他者のデータにアクセスさせない | 必須 |
| 認証 | MF の2FA（Authenticator/TOTP）を使用しているため、クラウドでの自動ログインは行わない | 必須 |
| 実行環境 | Windows 11、Chrome を前提。設定ファイルで User Data Dir と profile を必須指定（未指定はエラー） | 必須 |

## 設定ファイル仕様（概要）

設定ファイルの詳細は「設定ファイル YAMLスキーマ仕様書.md」を参照。

- **ファイル名・場所**：ツールフォルダ直下 `config.yml`
- **フォーマット**：YAML
- **必須項目**：`chrome_user_data_dir` / `chrome_profile` / `dry_run`（未記載で起動エラー）
- **任意項目**：`mapping_rules` / `exclude_prefixes` / `duplicate_detection` / `parser` / `log_settings` / `advanced`

## 検証方法（受入基準）

テストの詳細手順は「テストケース詳細手順書.md」を参照。

| テスト種別 | 項目 | 合格基準 |
| ----------- | ------ | --------- |
| 単体 | CSVパース（通常支払） | 全カラムを正しく抽出・数値化できること |
| 単体 | CSVパース（複合支払） | 金額が合算されて1件として処理されること |
| 単体 | CSVパース（ポイント入金） | 出金金額なし・入金金額ありの行が正しく処理されること |
| 単体 | 除外ルール（PPCD_A_） | 取引番号が PPCD_A_ で始まる行が出力に含まれないこと |
| 単体 | 重複検知 | 同一取引番号の2回目がスキップされること |
| 単体 | 文字コード | UTF-8 / Shift_JIS 両方で文字化けなく読み込めること |
| 結合 | ドライラン | ブラウザ起動・遷移が行われるが MF への書き込みが発生しないこと |
| 結合 | ログ出力 | logs_dir にログ・エラーCSV・スクショが正しく保存されること |
| 結合 | 設定バリデーション | 必須項目欠落時に起動エラーが出ること |
| E2E | フル自動実行 | MF に取引が正しく登録されること（ユーザー目視確認） |
| 障害 | DOM変化シミュレーション | セレクタエラー時にスクショ・ログが保存され中断されること |
| 異常系 | 欠損カラム | 不正行がエラーCSVに出力され他行は処理継続すること |

## 実行フロー（ユーザー操作）

1. Chrome を完全終了する。
2. ツール（EXE）を起動する（`config.yml` がツールフォルダ直下に存在し、必須項目が設定済みであることを確認）。
3. CSVファイルを選択して「ドライラン」または「実行」を選択する。
4. ツールが Chrome を指定の `user_data_dir` / `profile` で起動し、MFへ接続して自動でアップロード・登録を行う（またはドライランでシミュレーション）。
5. 実行結果（成功件数・除外件数・重複件数・失敗件数）をログとエラーCSV／スクリーンショットで出力する。
