# テストケース詳細手順書

## 概要

本ドキュメントは、PayPay→MoneyForward 自動登録ツールの受入テストケースおよびチェックリストを定義する。テストは「単体テスト」「結合テスト」「スモークテスト」「E2Eテスト」「障害・異常系テスト」の5層で構成する。

## テスト環境前提条件

| 項目 | 内容 |
| ------ | ------ |
| OS | Windows 11 |
| ブラウザ | Google Chrome（最新版） |
| Python | 3.9 以上 |
| ライブラリ | Playwright, PyYAML（インストール済み） |
| Playwright ブラウザ | `playwright install chromium` 実行済み |
| config.yml | ツールフォルダ直下に配置済み（必須項目が設定済みであること） |

## TC-01：設定ファイルロードテスト（単体）

**目的**：`config.yml` の存在チェック・バリデーション・デフォルト値補完が正しく動作することを確認する。

### TC-01-01：正常ロード

- [ ] `config.yml` に必須3項目（`chrome_user_data_dir` / `chrome_profile` / `dry_run`）が全て記載されている。
- [ ] ツールを起動したときエラーなく起動することを確認。
- [ ] ログに「config loaded」等の成功メッセージが出力されることを確認。

### TC-01-02：必須項目欠落（chrome_user_data_dir）

- [ ] `config.yml` から `chrome_user_data_dir` を削除してツールを起��する。
- [ ] 起動が中止されることを確認。
- [ ] エラーメッセージに `chrome_user_data_dir が設定されていません` 等の説明が含まれることを確認。

### TC-01-03：必須項目欠落（chrome_profile）

- [ ] `config.yml` から `chrome_profile` を削除してツールを起動する。
- [ ] 起動が中止されることを確認。
- [ ] 上記と同様のエラーメッセージが出力されることを確認。

### TC-01-04：必須項目欠落（dry_run）

- [ ] `config.yml` から `dry_run` を削除してツールを起動する。
- [ ] 起動が中止されることを確認。
- [ ] エラーメッセージに `dry_run が設定されていません` 等の説明が含まれることを確認。

### TC-01-05：必須項目欠落（input_csv）

- [ ] `config.yml` から `input_csv` を削除してツールを起動する。
- [ ] 起動が中止されることを確認。
- [ ] エラーメッセージに `input_csv が設定されていません` 等の説明が含まれることを確認。

### TC-01-06：必須項目欠落（mf_account）

- [ ] `config.yml` から `mf_account` を削除してツールを起動する。
- [ ] 起動が中止されることを確認。
- [ ] エラーメッセージに `mf_account が設定されていません` 等の説明が含まれることを確認。

### TC-01-07：logs_dir 未指定（デフォルト動作）

- [ ] `log_settings.logs_dir` を `null` または未指定にしてツールを起動する。
- [ ] `<tool_folder>\logs` が自動作成されることを確認。
- [ ] ログファイルが `<tool_folder>\logs` に出力されることを確認。

### TC-01-08：logs_dir 指定（任意パス）

- [ ] `log_settings.logs_dir` に任意の既存パス（例: `C:\temp\mf_logs`）を設定してツールを起動する。
- [ ] 指定したパスにログが出力されることを確認。

### TC-01-09：存在しない chrome_user_data_dir

- [ ] `dry_run: false` で `chrome_user_data_dir` に存在しないパスを設定してツールを起動する。
- [ ] 本番実行では起動が中止されることを確認。
- [ ] エラーメッセージにパスが存在しない旨が含まれることを確認。
- [ ] `dry_run: true` に変更して同じ設定で再実行する。
- [ ] ドライランでは Chrome パス検証がスキップされ、設定読み込みが成功することを確認。

## TC-02：CSVパーステスト（単体）

**目的**：CSVの各パターンが正しくパースされることを確認する。

### TC-02-01：通常支払（単一支払手段）

- 対象行：`04639628474580213761`（モスのネット注文、920円）
- [ ] 取引日が `2025-02-11 19:24:02` としてパースされることを確認。
- [ ] 出金金額が数値 `920` に変換されることを確認。
- [ ] 取引先が `モスのネット注文` として取得されることを確認。
- [ ] mapping_rules の `モス` ルールが合致し、カテゴリが `外食` となることを確認。

### TC-02-02：金額文字列の数値化（カンマ含む）

- 対象行：`PPCD_A_2025021122321300218846`（1,280円）
- [ ] 出金金額が `"1,280"` から数値 `1280` に正しく変換されることを確認（除外ルールのテストではなくパース処理の確認）。

### TC-02-03：複合支払（ポイント＋カード）

- 対象行：`04638686270424956930`（キャンドゥ、合計330円）
- [ ] 出金金額が `330`（ポイント73円＋カード257円）に合算されることを確認。
- [ ] 1件として処理されることを確認。
- [ ] mapping_rules の `キャンドゥ` ルールが合致し、カテゴリが `雑貨／日用品` となることを確認。

### TC-02-04：ポイント入金（出金なし）

- 対象行：`856574761326657536-a0196d18`（giftee、入金120円）
- [ ] 入金金額が `120` として取得されることを確認。
- [ ] 出金金額が `0` または `null` として扱われることを確認。
- [ ] mapping_rules の `giftee` ルールが合致し、カテゴリが `ポイント・ギフト` となることを確認。

### TC-02-05：文字コード（UTF-8）

## TC-04：重複検知テスト（単体／結合）

**目的**：同一取引が重複登録されないことを確認する。

### TC-04-01：取引番号による重複検知

- [ ] `test_input.csv` を1回目に処理し、ログに処理済み取引番号が記録されることを確認。
- [ ] 同一の `test_input.csv` を2回目に処理する。
- [ ] 2回目はすべての行が「重複スキップ」となることを確認。
- [ ] ログに `重複スキップ: N件` と記録されることを確認。

### TC-04-02：取引番号欠損時のフォールバック

- [ ] 取引番号が空の行を含むテストCSVを用意する。
- [ ] 1回目の処理後に同一行（日時・金額・取引先が同じ）を2回目に処理する。
- [ ] 2回目が重複スキップされることを確認。

## TC-05：ドライランテスト（単体）

**目的**：`dry_run: true` の場合にブラウザが起動せず、CSV 診断結果のみが出力されることを確認する。

### TC-05 前提条件

- `config.yml` の `dry_run: true` であること。
- `input_csv` に `test_input.csv` のパスが設定されていること。
- Chrome の起動状態は問わない（ドライランは Chrome を使用しない）。
- `chrome_user_data_dir` と `chrome_profile` は存在しなくてもよい（ドライランでは検証対象外）。

### TC-05-01：ドライラン実行

- [ ] ツールを起動する。
- [ ] Chrome が起動しないことを確認。
- [ ] 標準出力またはログに `DRY RUN: ブラウザを起動しません。CSV診断のみ実行します。` が出力されることを確認。
- [ ] ログに機微情報のローカル保管と非共有を促す warning が出力されることを確認。
- [ ] 処理件数・除外件数・重複スキップ件数が出力されることを確認。
- [ ] 個別取引の日付・金額・取引先が app ログへ出力されないことを確認。
- [ ] PPCD_A_ の取引が診断結果に含まれないことを確認。
- [ ] ドライラン実行後に同一 CSV を本番実行しても、ドライラン結果だけで全件が重複スキップされないことを確認。

### TC-05-02：ドライランのログ出力

- [ ] `<tool_folder>\logs`（または指定の logs_dir）にログファイルが出力されることを確認。
- [ ] ログに処理件数・除外件数・重複スキップ件数が含まれることを確認。
- [ ] ログに `input_csv` の絶対パスが含まれないことを確認。
- [ ] エラーCSVおよびスクリーンショットが生成されないことを確認（ドライランはエラーが発生しないため）。
- [ ] `processed.json` が新規作成されないこと、または既存の重複履歴が変化しないことを確認。

## TC-07：E2E テスト（本番実行）

**目的**：フル自動で PayPay CSV から MF に取引が正しく登録されることを確認する。

> ⚠️ 本テストは実際の MF アカウントに書き込みを行う。テスト前に MF 側でテスト確認しやすい状態（過去の取引が少ない期間等）を準備することを推奨する。

### TC-07 前提条件

- Chrome を完全終了していること。
- Chrome プロファイルで MF にログイン済みであること（Cookieが有効）。
- `config.yml` の `dry_run: false` であること。

### TC-07-00：Playwright スモークテスト（手入力モーダル起動確認）

- [ ] Chrome プロファイルが Money Forward ME にログイン済みであることを確認する。
- [ ] PowerShell で `PAYPAY2MF_RUN_SMOKE_TEST=1` を設定する。
- [ ] `PAYPAY2MF_SMOKE_CHROME_USER_DATA_DIR`、`PAYPAY2MF_SMOKE_CHROME_PROFILE`、`PAYPAY2MF_SMOKE_MF_ACCOUNT` を設定する。
- [ ] `python -m pytest -m smoke_test tests/test_mf_smoke.py` を実行する。
- [ ] Money Forward の入出金ページへ遷移し、手入力モーダルが開けることを確認する。
- [ ] 取引の保存送信が行われていないことを確認する。
- [ ] 通常の `pytest` 実行ではこのスモークテストが自動実行されないことを確認する。

### TC-07-00A：カテゴリ対応表 override 確認

- [ ] `config.yml` の `advanced.mf_categories_path` にテスト用 YAML パスを設定する。
- [ ] テスト用 YAML が `middle_to_large:` ルートを持ち、対象の中項目を任意の大項目へ変更していることを確認する。
- [ ] ツールを実行し、対象取引のカテゴリ選択でテスト用 YAML の対応が使用されることを確認する。
- [ ] `advanced.mf_categories_path` を `null` または未指定へ戻し、同梱定義へ復帰することを確認する。

### TC-07-01：正常登録

- [ ] Chrome を完全終了し `config.yml` の `dry_run: false` を設定してツールを起動する。
- [ ] ツールが MF のページへ遷移し、取引登録操作を自動実行することを確認。
- [ ] ツールの完了ログに成功件数・除外件数・重複スキップ件数・失敗件数が出力されることを確認。
- [ ] MF の取引一覧を目視確認し、PPCD_A_ 以外の取引が追加されていることを確認。
- [ ] MF の取引一覧に PPCD_A_ で始まる取引が追加されていないことを確認。

### TC-07-02：重複防止（E2E）

- [ ] TC-07-01 に続けて同一 CSV を再度実行する。
- [ ] 全件が重複スキップされることを確認。
- [ ] MF の取引一覧に追加が発生しないことを確認。

## TC-08：障害テスト（DOM 変化シミュレーション）

**目的**：MF 側の画面構造が変化した場合に、ツールが適切にエラー処理を行うことを確認する。

### TC-08-01：セレクタ未検出

- [ ] `advanced.screenshot_on_error: true` を明示し、テスト用に Playwright のセレクタを無効なものに変更してツールを実行する。
- [ ] ツールが処理を中断することを確認。
- [ ] logs_dir にスクリーンショット（エラー時）が保存されることを確認。
- [ ] スクリーンショットのファイル名に加盟店名が含まれないことを確認。
- [ ] ログに「セレクタが見つかりません」等の詳細エラーが出力されることを確認。
- [ ] ユーザーに中断原因と再実行手順が表示されることを確認。

## TC-09：異常系 CSV テスト（単体）

**目的**：不正な入力データが含まれる場合に適切に処理されることを確認する。

### TC-09-01：欠損カラム

- [ ] 「取引日」カラムが欠損した行を含む CSV を用意して実行する。
- [ ] 欠損行が解析エラーCSV（`parse_error_*.csv`）に出力されることを確認。
- [ ] `parse_error_*.csv` の列が `row_index,error_type,error_message` のみであることを確認。
- [ ] 他の正常な行が引き続き処理されることを確認。

### TC-09-03：登録失敗 CSV の最小化

- [ ] 登録途中で 1 件以上の失敗を発生させる。
- [ ] `error_*.csv` が出力されることを確認。
- [ ] `error_*.csv` の列が `failure_index,error_message` のみであることを確認。
- [ ] app ログに個別取引の日付・金額・加盟店が出力されていないことを確認。

### TC-09-02：金額が「-」（入金のみ）

- [ ] 出金金額が `"-"` の行を含む CSV を実行する。
- [ ] 出金金額が `0` または `null` として扱われ、処理が正常継続されることを確認。

### TC-09-03：取引番号が空

- [ ] 取引番号が空の行を含む CSV を実行する。
- [ ] 重複判定が「日時＋金額＋取引先」のフォールバックに切り替わることを確認。

## 受入基準（全体）

以下の全てを満たすことを受入条件とする。

| 基準 | 内容 |
| ------ | ------ |
| 設定バリデーション | 必須項目欠落時に100%起動エラーが発生すること |
| CSV変換精度 | テストサンプル全行中99%以上が期待どおりに変換されること |
| 除外ルール | PPCD_A_ で始まる取引が出力に含まれないこと（0件） |
| 重複防止 | 同一CSV2回実行で2回目が100%スキップされること |
| ドライラン | dry_run: true 時にブラウザが起動せず CSV 診断結果のみが出力されること |
| E2E登録 | MF に取引が正しく登録されること（ユーザー目視確認） |
| 障害ハンドリング | エラー時にスクショ・ログが保存され中断原因が表示されること |
| 異常系 | 不正行がエラーCSVに出力され処理が継続されること |
